# 分散システムにおけるデータ整合性確保の仕様 (Distributed Data Consistency Specification)

## 1. 概要
本ドキュメントは、RogueB におけるマイクロサービス間（Dungeon, Objects, BookOfAdventure 等）でのデータ整合性を確保するための技術仕様を定義します。本プロジェクトでは、パフォーマンスとスケーラビリティを重視し、重量級の分散トランザクション（2PC 等）に頼らず、結果整合性（Eventual Consistency）に基づいた整合性確保を目指します。

## 2. 冪等な API 設計 (Idempotent API Design)
ネットワークの不具合やリトライによる二重処理を防ぐため、状態を更新するすべての API は「冪等（Idempotent）」である必要があります。

### 2.1 要求識別子 (Request ID) の導入
- **仕様**: クライアント（または BFF である PlayerOperations）は、更新リクエストごとに一意な `requestId` (UUID) を生成し、HTTP ヘッダー (`X-Request-ID`) またはリクエストボディに含めます。
- **サーバー側の挙動**:
    1. リクエスト受信時、`requestId` が処理済みかキャッシュ（または DB）を確認します。
    2. 処理済みの場合は、前回と同じレスポンスを即座に返し、実際の更新処理はスキップします。
    3. 未処理の場合は、処理を実行し、結果を `requestId` と共に保存します。
- **有効期限**: 完了した `requestId` の履歴は、一定期間（例: 10分）経過後に自動的に削除されます。

## 3. Outbox パターンの採用 (Outbox Pattern)
データベースの更新と、他サービスへのイベント通知（メッセージ送信）をアトミックに実行するための仕組みです。

### 3.1 アウトボックステーブルの構造
各サービスのデータベースに `outbox` コレクション（またはテーブル）を設けます。
- `id`: 一意なイベント ID
- `aggregateId`: 対象となるエンティティの ID（プレイヤー ID 等）
- `type`: イベントの種類（`ITEM_PICKED_UP`, `LEVEL_UP` 等）
- `payload`: JSON 形式のイベントデータ
- `status`: 配信状態（`PENDING`, `PROCESSED`, `FAILED`）
- `createDate`: 作成日時

### 3.2 処理フロー
1. **更新フェーズ**: ビジネスロジックの実行に伴う DB 更新と、`outbox` へのイベント挿入を、同一のローカルトランザクション内で実行します。
2. **配信フェーズ**: 別スレッドまたは専用のワーカーが `outbox` から `PENDING` 状態のイベントを読み込み、対象のサービス（またはメッセージブローカー）へ送信します。
3. **完了フェーズ**: 送信成功後、`outbox` のステータスを `PROCESSED` に更新します。

## 4. 補償アクション (Compensation Actions)
複数のサービスにまたがる処理の一部が失敗した場合に、すでに行われた処理を取り消す、あるいは整合性を取るための「逆方向の処理」を定義します。

- **例: アイテム購入フロー**
    1. `BookOfAdventure`: ゴールドを減らして「購入中」ステータスにする。
    2. `Objects`: アイテムの所有権を変更する。 -> **失敗！**
    3. **補償処理**: `BookOfAdventure` に通知し、減らしたゴールドを戻してステータスを復旧する。

## 5. データ整合性チェック（リコンサイル） (Data Reconciliation)
システム的なエラーや予期せぬ不具合によって発生した不整合を、事後的に検知・修正するためのプロセスです。

### 5.1 定期バッチ処理
1. **インベントリ突合**: `BookOfAdventure` の `objectIdList` と、`Objects` モジュール側の所有権データを定期的に突合し、どちらか一方にしか存在しない「浮いたデータ」がないか確認します。
2. **不整合の修正**:
    - **Objects にのみ存在**: 所有権を「地面（Floor）」に戻すか、所有者に再紐付けします。
    - **BookOfAdventure にのみ存在**: 実体がないアイテム ID をリストから削除します。

### 5.2 整合性レポート
不整合が検知された場合、管理者に通知され、不整合の発生頻度や原因調査に利用されます。

## 6. 実装上の優先順位
1. **冪等性の確保**: 重複更新を防ぐための `requestId` 導入を最優先します。
2. **結果整合性の許容**: 数秒程度の遅延であれば、ユーザー体験を損なわない範囲で結果整合性を許容する設計にします。
3. **リコンサイルツールの作成**: 開発初期段階から、不整合を可視化するツールを整備します。
