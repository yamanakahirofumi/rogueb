# MongoDB と RDB の比較ガイド

本ドキュメントでは、マイクロサービスにおけるデータ保管先の選定基準とするため、MongoDB（NoSQL）とリレーショナルデータベース（RDB）の特性を中立的な視点で比較します。

## 1. 概要

### MongoDB (NoSQL)
ドキュメント指向のデータベースであり、データを JSON に似た BSON 形式で保持します。スキーマレスな性質を持ち、データの柔軟性と水平スケーラビリティに強みがあります。

### RDB (Relational Database)
MySQL や PostgreSQL に代表される、表形式（テーブル）でデータを管理するデータベースです。厳格なスキーマ定義とテーブル間のリレーションシップ（外部参照）、および強力な ACID トランザクションに強みがあります。

## 2. データモデルの比較

| 項目 | MongoDB | RDB |
| :--- | :--- | :--- |
| **基本単位** | ドキュメント（JSON/BSON） | 行（レコード） |
| **格納構造** | コレクション | テーブル |
| **データの関連付け** | 埋め込み（内包）または参照 | 結合（JOIN） |
| **柔軟性** | 非常に高い（階層構造をそのまま持てる） | 低い（正規化が必要） |

- **MongoDB**: 関連するデータを一つのドキュメントに埋め込むことで、一度のクエリで必要な情報を取得できます。
- **RDB**: データを正規化して複数のテーブルに分割し、結合（JOIN）を用いて整合性を保ちながらデータを取得します。

## 3. スキーマ設計

- **MongoDB (Dynamic Schema)**:
  - 事前の定義なしにフィールドを追加でき、ドキュメントごとに異なる構造を持つことが可能です。
  - アプリケーションの進化に合わせてデータ構造を柔軟に変更できますが、整合性の維持はアプリケーション側の責任が大きくなります。
- **RDB (Static Schema)**:
  - 事前にテーブル定義（DDL）が必要です。データ型や制約が厳格に守られます。
  - 構造変更には `ALTER TABLE` などのマイグレーション作業が必要になりますが、データベースレベルでデータの品質が保証されます。

## 4. スケーラビリティ

- **MongoDB (Horizontal Scaling)**:
  - **シャーディング**（データの分散配置）により、書き込み・読み込み負荷を複数のサーバーに分散させることが容易です。
  - ビッグデータや高トラフィックな環境に適しています。
- **RDB (Vertical Scaling)**:
  - 基本的には単一サーバーのスペックアップ（CPU, RAM）で性能を向上させます。
  - 読み込みに関してはレプリケーションによる分散が可能ですが、書き込みの水平分散は非常に複雑になります。

## 5. トランザクションと整合性

- **MongoDB**:
  - 単一ドキュメント内での操作は原子性が保証されます。
  - 近年のバージョンでは複数ドキュメントにまたがる ACID トランザクションもサポートされていますが、RDB ほど高速・軽量ではありません。
- **RDB**:
  - 強力な **ACID トランザクション**をサポートします。
  - 複数のテーブルにまたがる複雑な更新処理においても、完全な整合性を保証することが得意です。

## 6. パフォーマンスとクエリ

- **MongoDB**:
  - 埋め込み構造により JOIN を回避できるため、特定のドキュメントに対する読み書きが非常に高速です。
  - 複雑な集計処理（Aggregation Framework）も備えています。
- **RDB**:
  - インデックスが高度に最適化されており、構造化されたデータに対する複雑な検索や結合処理において高いパフォーマンスを発揮します。
  - 大量データの結合が多発すると、パフォーマンスが低下する傾向があります。

## 7. 代表的なユースケース

### MongoDB が適している場合
- データ構造が頻繁に変更される、または不確定な場合（カタログ、メタデータ管理）。
- 階層構造を持つデータをそのまま扱いたい場合（ログ、ソーシャルメディアの投稿）。
- 低遅延での大量な書き込みが必要な場合。
- 将来的な水平スケーリングが前提となっている場合。

### RDB が適している場合
- データの整合性と一貫性が極めて重要な場合（金融システム、決済、在庫管理）。
- データ構造が明確に定義されており、変化が少ない場合。
- 複雑なリレーションシップを持ち、多角的な分析クエリが必要な場合。
- 既存の SQL エコシステムや BI ツールとの連携を重視する場合。

## 8. 比較まとめ表

| 比較項目 | MongoDB (NoSQL) | RDB (Relational) |
| :--- | :--- | :--- |
| **スキーマ** | 動的（フレキシブル） | 静的（厳格） |
| **スケーリング** | 水平方向（安価なサーバー増設） | 垂直方向（高価な単一サーバー） |
| **トランザクション** | ACID (限定的 / 複数ドキュメント可) | ACID (強力 / 広範囲) |
| **JOIN** | 基本的には行わない（埋め込み） | 標準的（結合） |
| **開発スピード** | 速い（スキーマ変更が容易） | 普通（マイグレーションが必要） |
| **整合性** | 結果整合性（BASE）に近い運用も可 | 強固な一貫性 |

## 9. 併用（ポリグロット・パーシステンス）の検討

マイクロサービスアーキテクチャでは、サービスごとに最適なデータベースを選択する **「ポリグロット・パーシステンス（Polyglot Persistence）」** という考え方が一般的です。単一のデータベースですべてを解決するのではなく、MongoDB と RDB を併用することで、それぞれの利点を最大限に引き出すことができます。

### 代表的な併用パターン
- **マスタデータとトランザクションは RDB、非構造化データは MongoDB**:
  - ユーザー情報や決済データなど、厳格な整合性が求められるコアデータは RDB で管理します。
  - 一方で、商品カタログ、ユーザーの行動ログ、通知履歴など、データの種類が多様で頻繁に拡張されるデータは MongoDB で管理します。
- **書き込み負荷の高いデータは MongoDB、複雑な集計は RDB**:
  - IoT デバイスからのセンサーデータや、ゲームのリアルタイムな状態変更など、高頻度な書き込みには MongoDB を使用します。
  - そのデータを定期的に RDB に同期・集約し、経営分析や複雑なレポート作成に使用します。
- **CQRS（コマンド・クエリ責務分離）パターン**:
  - 書き込み（Command）側には整合性に強い RDB を使用し、参照（Query）側には検索性能やスケーラビリティに優れた MongoDB（または全文検索エンジン）を使用して、用途に応じてデータを最適化します。

### 併用時の留意点
- **運用コストの増大**: 複数のデータベースエンジンを管理・監視する必要があり、インフラ運用の複雑さが増します。
- **データの一貫性管理**: データベース間でのデータの同期（ETL 処理やイベント駆動による連携）が必要になり、システム全体としての「最終的な一貫性（Eventual Consistency）」を許容する設計が求められます。
- **学習コスト**: 開発チームが両方のデータベースの特性やクエリ言語（SQL と MongoDB Query Language）を理解している必要があります。
