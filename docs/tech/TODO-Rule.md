# TODOリストの書き方ルール

本プロジェクトにおいて、機能追加や技術的な課題を管理するための「TODOリスト（検討事項）」の記述ルールを定義します。

## 1. 管理場所
- メインの管理ファイル: `docs/TODO-Details.md`
- 関連ドキュメントからの参照: `docs/README.md` 等から上記の詳細ファイルへリンクを貼る。

## 2. 記述項目と構成

TODO項目は以下の構成で記述します。

### 2.1 未完了の項目
- **チェックボックス**: `[ ]` を使用。
- **タイトル**: 検討が必要な機能や課題の名称。
- **概要**: なぜその検討が必要なのか、解決すべき問題の要約。
- **検討内容**: 具体的にどのような点を決める必要があるか、または調査すべき事項を箇条書きで記載。

例：
```markdown
### [ ] 〇〇システムの設計
- **概要**: プレイヤーの〇〇アクションに伴う計算負荷の抑制。
- **検討内容**:
    - アルゴリズムの選定。
    - データの保持形式。
```

### 2.2 完了済みの項目
- **チェックボックス**: `[x]` に更新。
- **解決策**: 仕様が策定された場合、その仕様書（Markdownファイル）へのリンクを明記する。

例：
```markdown
### [x] 〇〇システムの設計
- **概要**: プレイヤーの〇〇アクションに伴う計算負荷の抑制。
- **解決策**: [〇〇システム仕様](../features/XX-System.md) にて、〇〇アルゴリズムの使用を策定済み。
```

## 3. カテゴリ分け
TODOリストは以下のカテゴリに分類して管理します。
1. **技術面**: パフォーマンス、マルチスレッド、データ構造など。
2. **ゲームデザイン面**: バイオーム、進行ガイド、クラフトレシピなど。
3. **アセット・リソース面**: スプライト, SE/BGM, UIデザインなど。
4. **実装時の詳細事項**: 開発中に見つかった、より踏み込んだ技術的課題。
5. **人間が主導すべき設計・調整面**: AIでは作成困難なゲームデータ、バランス、クリエイティブ要素。

## 5. 人間が考えるべき事項の取り扱い
TODOリストには、AIが効率的に生成可能な「定型的な実装」だけでなく、人間が「プレイ感覚」や「面白さ」を基準に判断・設計すべき項目を含めます。これらの項目には、検討内容に「人間が考えるべきこと」であることを明記してください。

- **AIが得意なこと**: コードの生成、既存パターンの拡張、定型的なリファクタリング、仕様に基づいたテスト作成。
- **人間が主導すべきこと**: ゲームバランスの心地よさ、操作感の微調整（手触り）、世界観の深化、全く新しいギミックの創出、プレイヤーの感情を動かす演出。

## 6. 更新のタイミング
- 新しい機能の構想が浮上したが、詳細な仕様が未定の時。
- 実装中に、現在の仕様では考慮不足な点（エッジケース等）を発見した時。
- 仕様が確定し、詳細ドキュメントが作成された時（ステータスを完了に更新）。
