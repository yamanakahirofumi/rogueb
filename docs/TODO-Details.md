## 1. 技術面
技術スタック、アーキテクチャ、パフォーマンスに関する検討事項です。

### [ ] モジュール間通信の最適化
- **概要**: 現在の REST ベースの通信におけるオーバーヘッドの評価と、必要に応じた最適化（gRPCや共有メモリの検討）。
- **検討内容**:
    - 現状のリアクティブスタックでのスループット計測。
    - サービス間でのデータ重複の排除。

### [ ] セーブデータの整合性確保（トランザクション以外の手法）
- **概要**: 複数サービス（Dungeon, Objects, BookOfAdventure）にまたがる状態更新時の整合性を、分散トランザクションに頼らずに確保する手法の検討。
- **検討内容**:
    - **不変（Idempotent）なAPI設計**: ネットワーク不具合等による再送が発生しても、二重更新を防ぐ仕組み（Request ID 等の導入）。
    - **結果整合性と補償アクション**: 一部のサービスで更新が失敗した際の、アプリケーションレベルでのクリーンアップ処理。
    - **データ整合性チェック（バッチ/リコンサイル）**: 定期的な不整合検知プロセス（例：インベントリと配置済みオブジェクトの突合）。
    - **Outboxパターンの検討**: DB更新とイベント発行の原子性を確保し、メッセージングによる後続処理の確実な実行。
